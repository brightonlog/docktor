# 깃 컨벤션 & MR 시 룰
## 1. 깃 컨벤션룰 

이것만 기억하세요. 저희의 깃 컨벤션 컨셉은 심플 그 자체입니다~!

소문자, 괄호 없이, : 쓰고, 뒤에 띄어쓰기 하고 내용 적으면 됩니다.

## 깃 컨벤션

| feat | ✨ 새로운 기능을 만들었을 때 |  |
| --- | --- | --- |
| fix | 🐛 버그를 고쳤을 때 |  |
| docs | 📝 문서를 수정했을 때 (README 등) |  |
| refactor | ♻️ 코드를 개선했을 때 (결과는 같지만 코드가 더 깔끔해짐) |  |
| style | 🎨 코드 의미에 영향 없는 스타일 수정 (세미콜론, 포맷팅 등) |  |
| test | ✅ 테스트 코드를 추가하거나 수정했을 때 |  |
| chore | 🔧 빌드 설정, 패키지 매니저 설정 등 잡다한 작업 |  |
| rename | 🚚 파일이나 폴더명을 바꿨을 때 |  |
| file, image | 🗂️ 파일을 추가했을 때 |  |


### 프롬프트

```
다음 규칙에 따라 깃 커밋 메시지를 작성해줘:

【규칙】
- 모든 글자는 소문자로 작성
- 괄호 사용 금지
- 형식: "타입: 메시지 내용" (콜론 뒤 띄어쓰기 1칸)
- 메시지는 간결하고 명확하게

【타입 목록】
- feat  → 새로운 기능 추가
- fix  → 버그 수정
- docs  → 문서 수정 (README 등)
- refactor  → 코드 개선 (기능 변화 없이 코드 품질 향상)
- style  → 코드 의미 변경 없는 스타일 수정 (포맷팅, 세미콜론 등)
- test  → 테스트 코드 추가/수정
- chore  → 빌드 설정, 패키지 관리 등 기타 작업
- rename  → 파일/폴더명 변경
- file, image  → 파일 추가

【작성할 내용】
'''
여러분의 작업 내용을 여기 적어주세요~~
```

- `feat: 로그인 화면에 소셜 로그인 버튼 추가`
- `feat: 소셜 로그인 기능 구현`
- `feat: 카카오, 구글 소셜 로그인 버튼 추가`


## 2. MR 시 규칙

###  MR Description 템플릿

```markdown
## 🎯 요약
- (무엇을 위한 작업인지 한 줄로 설명)

## 🛠 변경 사항
- (핵심 변경 로직이나 수정된 파일 목록 2~3개)
- 

## 👀 리뷰 포인트
- (리뷰어가 집중해서 봐줬으면 하는 부분이나 궁금한 점)`
```

### 프롬프트

```
너는 개발 팀의 효율적인 코드 리뷰를 돕는 전문 어시스턴트야. 
내가 제공하는 '코드 변경점(diff)'이나 '작업 내용'을 바탕으로 GitLab의 Merge Request(MR) 설명을 작성해줘.

**작성 규칙:**
1. 아주 간결하고 명확하게 핵심만 작성함.
2. 리뷰어가 짧은 시간 안에 핵심을 파악할 수 있도록 불필요한 미사여구는 제외함.
3. 아래 템플릿 형식을 엄격히 준수함.

**템플릿:**
## 🎯 요약
- (이번 작업의 핵심 목적을 한 줄로 요약)

## 🛠 변경 사항
- (주요 로직 변경이나 파일 수정 사항을 리스트 형태로 2~3개 작성)

## 👀 리뷰 포인트
- (리뷰어가 집중해서 확인해야 할 부분이나 피드백이 필요한 지점)

**입력 데이터 (코드 혹은 설명):**
[여기에 내용을 입력하세요]
```

# 3. Git Branch 규칙



```text
💡
1.  `master`와 개발용인 `develop` 브랜치를 나누어 사용합니다.
2. 각자 작업한 브랜치를 우선 `develop`에 합쳐서 기능들을 하나로 모읍니다
3. 모인 기능들이 최종적으로 확정되었을 때만, 최종적으로 `develop`을 `master`에 합칩니다.
4. 깨끗한  `master` 브랜치 유지 ^3^
```

## branch는 어떻게 적지?

**소문자**로, 파트/깃 컨벤선-세부기능 순으로 합시다!

브랜치 이름만 봐도 누가, 어디서, 무엇을 하는지 알 수 있게 적읍시다.

예시

- fe/feat-dashboard
- fe/fix-navbar
- be/feat-api-auth
- be/refactor-db-schema
